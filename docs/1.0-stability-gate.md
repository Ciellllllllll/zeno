# 1.0 Stability Gate

This gate defines the P0 SDK surface checks required before ZENO SDK can be considered for a later `v1.0.0` release phase.

It does not publish `v1.0.0`, create a GitHub Release, create a git tag, upload release assets, add gameplay samples, add engine features, redesign SDK/API/native/backend internals, or authorize committing generated SDK ZIPs or build outputs.

## Gate Inputs

The gate is based on:

- `docs/public-sdk-surface.md`
- `docs/sdk-compatibility-policy.md`
- `docs/renderer-backend-strategy.md`
- `docs/v1.0.0-readiness-criteria.md`
- `docs/pre-1.0-maturity-roadmap.md`
- `docs/sdk-layout.md`
- `docs/release-checklist.md`
- `docs/release-publication-runbook-v0.1.0-rc.1.md`
- package scripts and verification scripts under `scripts/`
- public headers under `sdk/cpp/include/zeno/` and `native/zeno_native/include/zeno/`

## Required Inventory Checks

Before any 1.0 release candidate can be promoted:

1. `include/zeno/` package contents must match the allowed header list in `docs/public-sdk-surface.md`.
2. No private implementation header may appear in packaged `include/zeno/`.
3. Every SDK-visible file or concept touched since the last RC must be classified as stable public SDK, experimental public SDK, demonstration-only sample surface, private implementation, or generated/build artifact.
4. Every stable public SDK addition, removal, rename, or signature change must be documented.
5. Every experimental public SDK change must be documented as experimental and must not be described as a mature 1.0 contract.
6. Every C ABI-facing change must preserve handle/POD/primitive/result-code rules.
7. Renderer backend changes must preserve backend-neutral public SDK and C ABI boundaries.
8. No generated/build artifact may be staged or committed.

## Required Compatibility Checks

Before promotion, the phase owner must confirm:

- public C++ SDK compatibility impact is understood
- C ABI compatibility impact is understood
- deprecations have replacement paths
- breaking changes are explicit and justified
- samples, templates, tutorials, and API docs are updated when affected
- `docs/release-notes.md` records public SDK or ABI changes since the previous candidate
- no public docs overclaim unsupported functionality
- DirectX 11 remains the default supported backend unless an explicit later phase changes that policy
- DirectX 12 is not described as complete until implementation and validation phases pass

## Required Verification Commands

The full future 1.0 readiness gate remains defined in `docs/v1.0.0-readiness-criteria.md`.

The SDK surface stability subset should include at least:

```powershell
.\scripts\verify-abi.ps1
.\scripts\verify-sdk-package-consumption.ps1
git diff --check
```

For a final 1.0 release decision, also run:

```powershell
.\scripts\verify-all.ps1
.\scripts\test-headless.ps1 -Configuration Debug
.\scripts\test-headless.ps1 -Configuration Release
.\scripts\verify-sdk-package-consumption.ps1
```

Phase56 is documentation-only, so these commands are defined here but not required to run in this phase.

## Release-Blocking Conditions

Any of these blocks promotion:

- packaged `include/zeno/` contains a header not listed in the public SDK surface inventory
- packaged `include/zeno/` contains a private implementation header
- C ABI exposes a forbidden C++/Rust/native platform type
- public SDK or C ABI exposes DirectX native objects
- game code is required to use DirectX-specific APIs
- DirectX 12 is claimed as complete before explicit implementation and validation phases
- default renderer backend changes away from DirectX 11 without an explicit approved phase
- C ABI struct layout, enum value, ownership rule, or result-code meaning changes without documentation
- stable public SDK item changes without compatibility notes
- experimental public SDK is presented as stable 1.0 API
- sample-only helper or gameplay behavior is described as SDK API
- generated SDK ZIP or build output is staged
- package consumption requires repository-internal source paths
- package text files include private absolute paths
- docs imply unsupported platform/backend/editor/scripting/networking/hot reload support
- release notes omit an SDK-visible change
- phase report omits public API / ABI change status

## Waiver Policy

A release-blocking SDK surface issue should be fixed, not waived.

A waiver is allowed only when:

- the issue does not violate C ABI safety
- the issue does not commit generated/build artifacts
- the issue does not publish, tag, or upload release assets
- the release owner records the reason in the phase report
- follow-up work is listed with a concrete future phase

Forbidden C ABI types, generated artifact commits, private header packaging, and accidental publication actions cannot be waived.

## Future Phase Update Procedure

When a future phase changes SDK-visible surface:

1. Update `docs/public-sdk-surface.md` if classification changes.
2. Update `docs/sdk-compatibility-policy.md` if compatibility rules change.
3. Update this gate if a new release-blocking condition or verification check is needed.
4. Update API docs, tutorials, package docs, and release notes when user-visible behavior changes.
5. Record Public API / ABI Changes in the phase report.

When a future phase does not change SDK-visible surface, state that explicitly in the phase report.
