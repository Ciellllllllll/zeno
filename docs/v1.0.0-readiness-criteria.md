# v1.0.0 Readiness Criteria

`v1.0.0` is not approved for publication yet.

This document defines objective readiness criteria for a later explicit 1.0 release decision. It does not publish a release, create a GitHub Release, create a git tag, upload release assets, or authorize committing generated SDK ZIPs or build outputs.

## Release Decision Rule

ZENO SDK `v1.0.0` may be considered ready only when all required criteria in this document pass and the result is recorded in a later phase report.

Passing these criteria still does not publish the release by itself. Publication must happen in a separate explicit release phase with explicit authorization to create the tag, GitHub Release, and uploaded assets.

## Required Criteria

### Scope And Identity

- `v1.0.0` scope is Windows first, MSVC first, and DirectX 11 first.
- Any unsupported platform, renderer, editor, scripting, networking, hot reload, installer, or production asset pipeline claim is absent from public docs.
- Release notes describe current capabilities and known limitations honestly.
- The SDK package identity, CMake version, release label, docs, workflow input, artifact name, future tag name, and future release title are consistent.

### Public SDK Contract

- Public C++ SDK headers under `include/zeno/` are intentionally listed and documented.
- Private implementation headers are not packaged.
- Public C ABI headers expose only handles, POD structs, primitive values, borrowed pointers with explicit lifetime rules, and `ZenResultCode`.
- No C++ class, `std::string`, `std::vector`, template, reference, exception, DirectX COM type, Win32 type, XAudio type, or Rust internal layout crosses the C ABI.
- Public API additions or removals since the last RC are documented in release notes.
- Public, experimental, and demonstration-only API areas are labeled.
- Compatibility, deprecation, and breaking-change rules are documented for post-1.0 SDK evolution.
- Public SDK support boundaries are explicit for `GameApp`, static modules, dynamic modules, resources, scenes, rendering, audio, input, diagnostics, and packaging.

### Lifecycle And Error Handling

- `GameApp` initialization, frame, render, shutdown, and partial-init failure behavior are documented and tested.
- Static-linked module lifecycle behavior is documented and tested.
- Dynamic module loading support boundary is documented and tested.
- `.zproj` and `.zscene` format rules, version fields, invalid cases, and compatibility expectations are documented.
- Invalid resource ID, stale resource ID, wrong resource type, missing asset, invalid scene, shader compile failure, and backend initialization failure cases return actionable `zeno::Result` values or C ABI result codes.
- Rust C ABI exports contain panic containment appropriate for a public 1.0 boundary.
- No recoverable runtime error relies on `panic!` or uncaught C++ exceptions across the ABI.

### Build And Test Gate

The 1.0 release gate must pass from a clean checkout on the intended Windows environment:

```powershell
.\scripts\verify-all.ps1
.\scripts\test-headless.ps1 -Configuration Debug
.\scripts\test-headless.ps1 -Configuration Release
.\scripts\verify-sdk-package-consumption.ps1
```

The gate fails if any command fails.

### Package Quality

- SDK package generation produces one expected top-level SDK directory.
- SDK ZIP structure is deterministic enough for documented inspection.
- Debug and Release `lib/` and `bin/` directories contain the expected artifacts.
- Packaged samples and templates consume the SDK through `find_package(ZenoEngine CONFIG REQUIRED)`.
- External package consumption does not use repository-internal include paths or source paths.
- Packaged docs include getting started, layout, API, tutorials, release notes, checklist, and readiness criteria.
- Package text files do not contain private local absolute paths.
- Generated SDK ZIPs, extracted package directories, build outputs, executables, DLLs, LIBs, PDBs, Cargo target outputs, and CMake build trees are not committed.

### Sample And Template Proof

- The packaged minimal template builds and runs one headless frame in Debug and Release.
- A packaged practical game template demonstrates the `GameApp` and static `GameModule` authoring path without repository-internal source paths.
- Focused 2D input/audio sample builds in Debug and Release.
- Focused 3D mesh sample builds in Debug and Release.
- Sample assets are present in the extracted SDK and copied runtime output directories.
- Windowed sample behavior has either stable automation or a repeatable manual visual inspection checklist with recorded results.
- If windowed visual or audio validation is waived for a candidate, the release owner records the explicit waiver and reason in that candidate's report.
- Samples demonstrate only supported public SDK workflows.

### Documentation Gate

- `README.md` and `docs/index.md` route users to the current SDK usage path.
- `docs/architecture.md` explains the Rust/C++/C ABI ownership split.
- `docs/sdk-guide.md` explains the current C++ SDK surface and lifecycle.
- `docs/sdk-layout.md` matches the generated package layout.
- `docs/api/` covers every advertised public API area.
- `docs/tutorials/` covers template startup and focused sample usage.
- `docs/release-notes.md` lists known limitations and blockers.
- `docs/pre-1.0-maturity-roadmap.md` and this document are updated before any 1.0 release decision.

### Release Infrastructure

- The existing RC artifact and publication workflow remains available.
- The release checklist has been run for the intended 1.0 candidate.
- The release artifact provenance is recorded.
- The SDK ZIP checksum is recorded.
- The artifact-only GitHub Actions workflow passes for the intended 1.0 package input, including Debug headless tests, Release headless tests, and SDK package consumption QA.
- After any later publication, downloaded release assets must match the recorded checksum, ZIP structure, and expected docs inventory.
- A later release publication runbook exists for the exact 1.0 candidate.
- The release owner confirms there is no pre-existing conflicting local or remote tag for the candidate.
- The release owner confirms there is no pre-existing conflicting GitHub Release for the candidate.

## Explicit 1.0 Blockers

- Any failing required release-gate command.
- Any undocumented public SDK or ABI surface change.
- Any forbidden C ABI type crossing the Rust/C++ boundary.
- Any generated SDK ZIP or build output staged for commit.
- Any private local path or repository-private source dependency inside the SDK package.
- Any release notes that imply unsupported platforms, renderers, editor tooling, scripting, networking, hot reload, or production asset pipeline support.
- Any unrecorded artifact provenance or missing checksum for the candidate.
- Any missing windowed/manual validation result or explicit waiver.
- Any missing later explicit authorization to publish, tag, or upload assets.

## Publication Boundary

When every criterion passes, the correct next step is not automatic publication.

The next step is a new explicit release phase that may authorize:

- creating the final `v1.0.0` or SDK-specific tag
- creating the GitHub Release
- uploading the verified SDK ZIP
- recording final post-publish checks

Until that phase exists, `v1.0.0` remains deferred.
